录音不只是文字。
它是下一步行动的开始。

一个 Claude Code skill：把实验室 / 办公室录音变成可审查的结构化产出 —— 摘要、按人拆分的任务、消息草稿。五种 preset 场景；不确定时主动询问、不胡猜；任何对外发送都先经人工审查。 

普通 ASR     → 音频 → 一大块文字（然后你坐下来读一晚上）

这个 skill   → 音频 → 摘要 + 任务 + 草稿 → 你审查 → 发送
▶ 试一下交互式 demo View on GitHub

它解决的问题

研究所 / 课题组里录音随处可见，但转写本身远远不够。

组会、师生 1:1 讨论、临时聊天、语音备忘录、进展汇报 —— 每种场景需要的处理都不一样。把组会模板套到临时聊天上，会得出"张同学需要在周五前完成 X"这种根本没人说过的"任务"。这是普通 LLM workflow 最常见的失败模式。

audio-to-action 给每种场景一个 preset，配一套 prompt 模板，明确什么能输出、什么必须先问用户。

不该做的事

音频 → 转写 → 强行套"组会摘要" → 编造学生姓名 → 编造截止时间 → 自动群发

每一步都在累积错误。最后那条群发是不可逆的。

这个 skill 做的事

音频 → 转写 → 清理 → 分类（5 种 + unknown） → 套对应 preset 或不确定时问用户 → 输出待审查产物 → 永远不自动发

每一步的不确定性都被显式标注。"推测"、"待确认"、"未明确"是常用词。

合并规则（transcript_normalizer.py::_merge_diarization）：每个 segment 计算与所有 diarization turn 的时间重叠，取重叠最长的那个作为 speaker 标签。重叠为 0 时保持 None —— 绝不就近赋值或猜测。

# tools/transcript_normalizer.py
def _merge_diarization(transcript, turns):
    for seg in transcript.segments:
        best_overlap, best_speaker = 0.0, None
        for start, end, spk in parsed_turns:
            overlap = max(0, min(seg.end, end) - max(seg.start, start))
            if overlap > best_overlap:
                best_overlap, best_speaker = overlap, spk
        if best_speaker and best_overlap > 0:
            seg.speaker = best_speaker   # overlap 为 0 时不赋
真实运行回到这里： 我们这条录音的 segments=[]，turns 也没有，所以 diarization=false。下游"按 speaker 拆分"的输出仍会照常生成 —— 用 SPEAKER_01 / 临时-N 占位符，并在不确定项里标"待确认收件人"。决定权交给用户，不替用户拒绝。

每个 preset 配一套 prompt 模板和必填字段。

group_meeting

组会

导师 + 多个学生，明确任务分派。输出含按人拆分的任务、待审查消息清单、待确认问题。

advisor_student_discussion

师生 1:1

导师与单个学生的讨论，含指导意见和下一步任务。输出含进展、建议、可发送消息草稿。

casual_discussion

临时讨论

非正式聊天，想法阶段。输出聚焦核心观点、达成共识、未解决问题，不强行编出任务。

voice_memo

语音备忘录

单人对自己说话。输出整理后的笔记 + Todo 列表 + 后续可展开方向。

student_progress_report

学生进展汇报

学生汇报上周工作。输出已完成事项、当前问题、需要导师反馈的点。

unknown

不确定

置信度太低或内容碎到无法归类时用的兜底标签。即便分到 unknown，skill 也会按 transcript 给出尽力输出 + A–F 菜单让用户决定方向。

一条硬规则 + 四条诚实规则

默认姿态是先服务，再标注。skill 的工作是把"录音里说了什么"和"我推测了什么"分开列清，把决定权交给用户 —— 不是替用户拒绝。

一条真正会拦动作的硬规则

  • !

    对外发送必须显式 confirm

    草稿生成 → 用户审查 → 用户点 发送 → 才出去。模型不能"决定发了"，因为发出去是不可逆的。这是 skill 唯一会真正拦住的动作；也是 STEP 5 选 E 时演示的那个流程。

四条规范怎么写、不阻断是否输出的诚实规则

  • ·

    不静默编造身份

    不知道叫谁就用 SPEAKER_01 / 临时-1，并在不确定项里标"待确认收件人"。"张同学"不会凭空冒出来。

  • ·

    推测必标注

    没说的截止时间、推断的归属、补的下一步行动 → 一律打 （推测） / （待确认）。让 reviewer 一眼分清"录音里说的"和"模型补的"。

  • ·

    不在个人草稿里默认夹带全文

    草稿默认只引相关 excerpt。用户主动要全文再给。

  • ·

    API key 走环境变量

    settings.yaml 写 api_key_env: ASR_API_KEY，不内联、不回显、不写日志。

三层架构，各自可换

ASR 与 LLM 后处理解耦；prompt 是纯 markdown。

Prompts 层 — prompts/*.md

LLM 行为契约。纯 markdown，可审查、可 diff、可 PR。改 prompt 不动代码。

Tools 层 — tools/*.py

数据契约，类型严格。dataclass 验证、provider 注册表、归一化、ffmpeg 包装、密钥脱敏。

配置层 — settings.yaml

切 ASR provider、调阈值、开关安全策略。不动代码。

cd audio-to-action

#    编辑 settings.yaml::asr.openai_compatible.base_url
export ASR_API_KEY=sk-...
