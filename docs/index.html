录音不只是文字。
它是下一步行动的开始。

一个 Claude Code skill：把实验室 / 办公室录音变成可审查的结构化产出 —— 摘要、按人拆分的任务、消息草稿。五种 preset 场景；不确定时主动询问、不胡猜；任何对外发送都先经人工审查。 

普通 ASR     → 音频 → 一大块文字（然后你坐下来读一晚上）

这个 skill   → 音频 → 摘要 + 任务 + 草稿 → 你审查 → 发送
View on GitHub → 看一段真实演示

它解决的问题

研究所 / 课题组里录音随处可见，但转写本身远远不够。

组会、师生 1:1 讨论、临时聊天、语音备忘录、进展汇报 —— 每种场景需要的处理都不一样。把组会模板套到临时聊天上，会得出"张同学需要在周五前完成 X"这种根本没人说过的"任务"。这是普通 LLM workflow 最常见的失败模式。

audio-to-action 给每种场景一个 preset，配一套 prompt 模板，明确什么能输出、什么必须先问用户。

不该做的事

音频 → 转写 → 强行套"组会摘要" → 编造学生姓名 → 编造截止时间 → 自动群发

每一步都在累积错误。最后那条群发是不可逆的。

这个 skill 做的事

音频 → 转写 → 清理 → 分类（5 种 + unknown） → 套对应 preset 或不确定时问用户 → 输出待审查产物 → 永远不自动发

每一步的不确定性都被显式标注。"推测"、"待确认"、"未明确"是常用词。

一次真实的端到端运行

用一段实验室真实录音的 60 秒片段走完整个 pipeline。下面所有 JSON、文本、用时数字均来自真实运行结果，未经修饰。

STEP 1

probe_audio — 文件预检

检查格式、大小、时长。本机没有 ffprobe，duration 字段降级为 None 并发出 warning，但不阻断流程。

# tools/utils.py::probe_audio  (real stdout)
[probe]    /tmp/asr_probe_60s.wav
[probe]    ext=wav  size=1.83 MB  duration=None
[probe]    warn: ffprobe not found on PATH; cannot determine duration.
                 Install ffmpeg to enable duration checks.
STEP 2

asr_client.transcribe — 调 ASR

完全走 OpenAI 兼容路径：POST /v1/audio/transcriptions。adapter 不知道服务端是 Doubao、Whisper 还是 vLLM —— 这是配置层决定的。

# settings.yaml::asr.openai_compatible
base_url:    http://127.0.0.1:8836/v1
model:       doubao-asr
language:    zh
api_key_env: ASR_API_KEY   # env var unset → 不发 Authorization 头
// raw response from Doubao
{
  "task":     "transcribe",
  "language": "zh",
  "duration": 0.0,
  "text":     "是测了什么？我们测了什么，对吧？我比别人更完备的在哪些地方？你自己这个表其实不换论文，你这个得有知道，要不然现在就是你不是一个按照论文的作文方式去做，因为现在就又随机又走了。嗯，我现在又变成了，那我。"
  // note: no "segments" key — this server doesn't emit them
}
关键发现： 这个 endpoint 不返回 segments 数组，只给 text。普通 ASR wrapper 到这里就结束 —— 但 skill 的下游会因此触发安全机制，拒绝生成需要时间戳引文的产物（per-person 草稿、按人拆分任务）。这不是 bug，是设计。
STEP 3

normalize — 归一化为统一 Transcript

无论 provider 给什么形状，下游只看见这一种 schema。segments 缺失时不抛错，留空数组，把判断留给后续步骤。

// out/transcript.json (real output)
{
  "source_file": "/tmp/asr_probe_60s.wav",
  "language":    "zh",
  "duration":    60.13,
  "text":        "是测了什么？...那我。",
  "segments":    [],          // ← empty (provider gave none)
  "metadata": {
    "asr_provider": "openai_compatible",
    "asr_model":    "doubao-asr",
    "diarization":  false
  }
}
STEP 4

classify — 这是哪种录音？

按 prompts/classify_audio_content.md 的契约，输出严格 JSON。下面是 LLM 应用该 prompt 后的真实输出。

{
  "label":        "casual_discussion",
  "confidence":   0.55,
  "rationale":    "单人监督式自语 + 抽象的论文写作方法学批评，无明确任务分派、无具体截止时间；介于'临时讨论'与'导师独白'之间，故置信度偏低。",
  "alternative_label":      "advisor_student_discussion",
  "alternative_confidence": 0.42
}
STEP 5

safety branch — 不确定时问，不要猜

置信度 0.55 < 阈值 0.6，且 segments 为空（无法引用时间戳）。skill 不硬套 preset，按 prompts/ask_followup.md 弹菜单：

我已完成转写（约 1m0s，1 段）。
这段录音看起来更像 一次临时讨论，置信度 0.55（偏低）。

你希望我怎么处理？

  A. 只整理全文转写
  B. 总结核心观点
  C. 提取待办事项
  D. 按参与人拆分任务
  E. 生成可发送给他人的消息草稿
     （**会先给你审查，不会自动发送**）
  F. 自定义处理方式（请告诉我具体目标）

附加问题（可选回答）：
  - 这段录音里只有一位说话人，还是有多人？
    我没拿到 diarization 数据。
  - 是否有保密要求？我可以避免把全文写入任何对外消息草稿。
注意一个隐含的拒绝： 即使你接下来选 D 或 E，由于 segments=[]，skill 仍会拒绝输出。prompts/extract_tasks.md 和 prompts/build_student_messages.md 都要求每条任务 / 草稿附时间戳引文 —— 这条硬规则在 prompt 和 settings 两层冗余，单靠"被说服"绕不过去。
STEP 6

假设用户选 B：总结核心观点

B 不依赖时间戳引文，可以执行。下面是 LLM 按 casual_discussion preset 输出的内容（节选）：

# 临时讨论整理（节选）

## 一、讨论背景

  说话人正在反思一项工作的"测了什么"以及"相比其他工作的完备性"，
  并就该工作对应论文的写作组织方式提出批评。

## 二、核心观点

  - "测了什么 / 我们测了什么"是核心问题：当前结果尚未清楚回答
    "我们与其他工作在哪些维度上更完备"。
  - 当前论文的写作组织偏松散，缺少一张明确的对比维度表，导致
    论述"随机又走了"，读者难以抓住主线。

## 四、未解决问题

  - 究竟用哪些维度作为"更完备"的衡量标准？录音中未明确。
  - 论文的当前写作框架是否需要重写？讨论提到了缺陷，未决断。

## 五、下一步行动 （建议，待确认）

  > 这部分是模型基于讨论内容的整理（推测），不是会议中明确
  > 交代的任务。是否纳入待办，请你确认。

  - （推测）先把"对比维度表"补齐再写正文，否则论述顺序容易回退。
  - （推测）把"我们和别人的完备性差异"作为论文 §1 的锚点。
叙事完整性： 输出里所有"建议 / 下一步"全部带 （推测） 标记 —— 因为录音原话只点出问题，没给出建议；下游读者很容易把模型补的建议当成"录音里的结论"，所以必须显式区分。

五个 preset，加一个"我不确定"

每个 preset 配一套 prompt 模板和必填字段。

group_meeting

组会

导师 + 多个学生，明确任务分派。输出含按人拆分的任务、待审查消息清单、待确认问题。

advisor_student_discussion

师生 1:1

导师与单个学生的讨论，含指导意见和下一步任务。输出含进展、建议、可发送消息草稿。

casual_discussion

临时讨论

非正式聊天，想法阶段。输出聚焦核心观点、达成共识、未解决问题，不强行编出任务。

voice_memo

语音备忘录

单人对自己说话。输出整理后的笔记 + Todo 列表 + 后续可展开方向。

student_progress_report

学生进展汇报

学生汇报上周工作。输出已完成事项、当前问题、需要导师反馈的点。

unknown

不确定

置信度 < 0.6 时不硬套模板，弹 A–F 菜单让用户选。这一项是这个 skill 的灵魂。

五条硬规则

在 prompt 和 settings 两层冗余实现，即使被说服也兜得住。

  • ✗

    不编造 speaker 身份

    SPEAKER_01 保持 SPEAKER_01，直到用户给出映射。"张同学"不会凭空出现。

  • ✗

    不编造截止时间、任务归属

    推测内容必须打 （推测） 标记，不能伪装成事实。

  • ✗

    不自动发送任何东西

    MVP 只生成草稿；发送是另一个明确审批的工具。output.send_messages: false 是配置硬开关。

  • ✗

    个人草稿里不夹带完整 transcript

    草稿只引用与本人有关的段落，不漏会议全貌。safety.forbid_full_transcript_in_drafts: true。

  • ✗

    没引文 ≠ 任务

    每条任务必须附时间戳引用。没有引文的"任务"自动降级为"待确认问题"。

三层架构，各自可换

ASR 与 LLM 后处理解耦；prompt 是纯 markdown。

Prompts 层 — prompts/*.md

LLM 行为契约。纯 markdown，可审查、可 diff、可 PR。改 prompt 不动代码。

Tools 层 — tools/*.py

数据契约，类型严格。dataclass 验证、provider 注册表、归一化、ffmpeg 包装、密钥脱敏。

配置层 — settings.yaml

切 ASR provider、调阈值、开关安全策略。不动代码。

5 分钟上手

假设你已经有 Claude Code 和一个 OpenAI 兼容的 ASR endpoint。

# 1. clone 到 Claude Code 的 skills 目录
git clone https://github.com/shcw0405/audio-to-action.git
cp -r audio-to-action ~/.claude/skills/

# 2. 安装运行依赖
cd audio-to-action
pip install -r requirements.txt

# 3. 指向你的 ASR endpoint
#    编辑 settings.yaml::asr.openai_compatible.base_url
export ASR_API_KEY=sk-...

# 4. 在 Claude Code 里丢一个音频文件
#    "整理一下这段录音 ./meeting.mp3"

批处理 / 调试场景可以直接用 Python 入口，不必走 Claude Code：

python scripts/run_pipeline.py path/to/audio.m4a \
    --settings settings.yaml \
    --out-dir ./out
